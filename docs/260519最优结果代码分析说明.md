## 1. 当前结论

当前代码中存在两条相关但不同的主线。

第一条是原始方案式的完整安全约束接口：`coop_safety/interface.py` 中的 `SafetyConstraintModule` 串联盲区推理、RiskMap、RiskGraph、RiskEvents、可行域收缩、长期可行性检查和最小伤害模式。它最接近 `研究点三方案说明.docx` 中“独立安全动作空间约束”的完整设定。

第二条是当前实验最优主线：`coop_safety/learned/hybrid_safety.py` 中的 `HybridSafetyConstraint` 以及 `experiments/run_modified_sumo_comparison.py` 中的 SUMO 包装变体。它不是完整三层可行域接口，而是“V1 神经检测 + 几何路点修正 + TTC/后车风险闭环控制”的实用化版本。当前 DeepAccident 和 SUMO 最优结果主要来自这条主线。

因此，现有方法最合理的论文叙事不是“完整三层框架已全部验证优于所有方法”，而是：

- 以独立安全约束为目标接口；
- 在代码中落地了完整三层风险-约束模块；
- 当前最优实验版抽取其中最有效的子机制，形成轻量的神经-几何安全约束；
- 在 DeepAccident 离线评测中证明路点碰撞风险最低；
- 在修改后的 SUMO 盲区和压力场景中证明 TTC、后车感知和横向逃逸能降低真实碰撞、二次碰撞和严重度。

## 3. 当前最优方法的模块与公式

### 3.1 输入接口

DeepAccident 和 SUMO 都被转换成统一的 `PerceptionResult`。每个 agent 包含相对位置、速度、尺寸、类型、可见性和置信度，结构定义见 `coop_safety/interface.py:32` 到 `coop_safety/interface.py:101`。

DeepAccident loader 使用 3D 标签构造 GT perception：`experiments/deepaccident_loader.py:55` 解析标签，`experiments/deepaccident_loader.py:91` 转成 `PerceptionResult`。默认 `include_invisible=True`，不可见目标也进入输入，但 `is_visible=False`、`confidence=0.3`。

SUMO perception 在 `experiments/run_modified_sumo_comparison.py:302` 构造。它先用遮挡车 `blk_*` 做 LOS 判断，再把可被 ego 或 coop 看到的车辆转换到 ego 坐标系。若目标只能被协同车看到，则 `is_visible=False`，但仍可进入本文方法的不可见目标安全边界。

### 3.2 V1 碰撞检测网络

V1 网络在 `coop_safety/learned/collision_network.py` 中实现：

- `AgentEncoderV2`：对每个 agent 的 10 维特征做 MLP 编码，见 `coop_safety/learned/collision_network.py:22`；
- `InteractionModule`：4 头 self-attention 建模 agent-agent 交互，见 `coop_safety/learned/collision_network.py:36`；
- `AttentionPool`：对 agent token 做注意力池化，见 `coop_safety/learned/collision_network.py:50`；
- `collision_head` 输出碰撞概率，`ttc_head` 输出 TTC，见 `coop_safety/learned/collision_network.py:75`。

输入特征为：

```text
x_i = [x, y, vx, vy, heading, length, width, speed, visible_flag, 0]
```

V1 输出：

```text
p_coll = sigmoid(MLP(pool(SelfAttention(MLP(x_i)))))
```

Hybrid 中只使用 `p_coll` 做预警判断，入口在 `HybridSafetyConstraint._detect_with_v1()`，见 `coop_safety/learned/hybrid_safety.py:157`。代码中 TTC head 被训练但 Hybrid 控制逻辑没有直接使用网络 TTC，这是当前实现和论文叙事需要说明的差异。

### 3.3 可见性与接近速度自适应安全边界

Hybrid 的核心几何安全边界在 `coop_safety/learned/hybrid_safety.py:91`：

```text
m_i =
  m_visible,   if agent is visible
  m_invisible, if agent is V2X-only / blind

v_app = - (r_i · v_rel_i) / ||r_i||

if v_app > 0:
  m_i = m_i * (1 + alpha * min(v_app / v_max, 1))

m_i = m_i + 0.3 * max(length_i, width_i)
```

默认参数来自构造函数：可见目标 2.5m、不可见目标 4.0m、接近速度系数 0.3、`v_max=20m/s`，见 `coop_safety/learned/hybrid_safety.py:59`。

该模块对应原始设定中的“协同感知 + 可见性/盲区范围”。区别是：完整方案会显式构造盲区和 phantom agent；Hybrid 直接把不可见但被协同感知补充的真实目标赋予更大 margin。

### 3.4 多智能体排斥式路点修正

对每个候选路点 `w_t`，Hybrid 假设 agent 匀速运动：

```text
a_i(t) = a_i(0) + v_i * ((t + 1) * 0.5)
```

若：

```text
||w_t - a_i(t)|| < m_i
```

则该 agent 是当前路点威胁，见 `coop_safety/learned/hybrid_safety.py:111`。

对所有威胁同时求合力，而不是只避最近目标：

```text
d_i = max(||w_t - a_i(t)||, 0.1)
q_i = (m_i - d_i + c) / d_i
lambda_i = 1.0 if visible else 1.5
F = sum_i lambda_i * (1 + risk_t) * q_i * (w_t - a_i(t)) / d_i
w'_t = w_t + normalize(F) * max_i(m_i - d_i + c)
```

代码见 `coop_safety/learned/hybrid_safety.py:126`。这里的 `risk_t` 预留给 V2 waypoint risk，但当前主线通常没有加载 `risk_model`，所以实际为 0。

路点修正之后做两步后处理：

- 平滑：`w_t = (1-beta)w_t + beta*(w_{t-1}+w_{t+1})/2`，见 `coop_safety/learned/hybrid_safety.py:257`；
- 速度限幅：相邻路点距离不超过 `v_max * 0.5`，见 `coop_safety/learned/hybrid_safety.py:267`。

该模块是 DeepAccident 上 WPC% 最低的主要原因。它和 MAP/UniE2EV2X 的差异是：MAP/UniE2EV2X 基本按单个碰撞目标处理，Hybrid 对所有威胁求合力，减少“避开一个目标却推向另一个目标”的情况。

### 3.5 检测和修正解耦

Hybrid 的关键工程设计写在 `coop_safety/learned/hybrid_safety.py:6` 和 `coop_safety/learned/hybrid_safety.py:230`：

- 路点修正始终由几何模块执行；
- 检测标志由 V1 概率决定；
- stats 中分别返回 `n_collisions_detected` 和 `n_geometric_threats`，见 `coop_safety/learned/hybrid_safety.py:316`。

这个设计带来两个效果：

- 路点安全不依赖神经网络是否触发，因此 WPC% 稳定较低；
- 预警可以通过 V1 阈值调节 false alarm，而不影响几何修正。

但这也是公平性问题：单一机制基线通常用同一套逻辑同时决定检测和修正，而 Hybrid 用两套独立机制分别优化两个指标。代码本身已在 `coop_safety/learned/hybrid_safety.py:10` 写了 fairness note。

### 3.6 TTC 速度控制

Hybrid 计算所有 agent 的最小 TTC，见 `coop_safety/learned/hybrid_safety.py:208`：

```text
approach_i = - (r_i · v_i) / ||r_i||
TTC_i = ||r_i|| / approach_i, if approach_i > 0.1
min_ttc = min_i TTC_i
```

速度因子决策树见 `coop_safety/learned/hybrid_safety.py:282`：

```text
if min_ttc < 2.0 or (n_threats >= 2 and p_coll > 0.5): factor = 0.1
elif min_ttc < 3.0 or n_threats >= 2 or p_coll > 0.7: factor = 0.3
elif min_ttc < 5.0 and (n_threats >= 1 or p_coll > thr): factor = 0.5
elif n_threats >= 1 or p_coll > thr: factor = 0.7
else: factor = 1.0
```

SUMO runner 把这个 factor 映射成闭环速度控制：`target_speed = ego_speed * factor`，见 `experiments/run_modified_sumo_comparison.py:630` 到 `experiments/run_modified_sumo_comparison.py:651`。

### 3.7 SUMO 压力场景改进

当前 SUMO 最优变体主要在 `experiments/run_modified_sumo_comparison.py`：

- `HybridWithGeometricANDTTC`：V1 与几何 AND，同时短 TTC 强制触发，见 `experiments/run_modified_sumo_comparison.py:95`；
- `RearAware`：区分前/侧 TTC 与后车 TTC，避免把后车追尾风险误转化为强制急刹，见 `experiments/run_modified_sumo_comparison.py:156`；
- `MinHarm`：当前/后方风险冲突时选择较低伤害的速度策略，见 `experiments/run_modified_sumo_comparison.py:190`；
- `RearEscape`：后车接近且前方威胁不立即致命时，保持速度并尝试横向换道，见 `experiments/run_modified_sumo_comparison.py:232` 和 `experiments/run_modified_sumo_comparison.py:536`。

RearEscape 的核心判断：

```text
close_rear = rear_gap < 18m or rear_ttc < 2.5s
if close_rear and front_ttc >= 1.0s:
    lane_escape = 1
    target_speed_factor = 1.0
elif front_ttc < 1.0s:
    lane_escape = 1 if close_rear else 0
    target_speed_factor = 0.5
```

这对应原始“最小伤害模式”的工程化版本：当完全刹停会造成后车冲击时，不追求绝对停下，而是选择较小总体风险的速度/横向动作。

## 7. 代码合理性与公平性自检

当前代码不能直接视为“无问题”。主要问题如下。

1. checkpoint AUC 与旧说明不一致。当前 `models/collision_net_best.pt` 记录 AUC=0.8142，不是 0.985。论文和文档必须以当前 checkpoint 为准。

2. Hybrid 是双机制方法。几何修正负责 WPC%，V1 负责检测/误报，这比单机制基线有结构优势。需要报告 ablation：no-net、no-geom、AND/TTC/RearEscape，而不是只报最终组合。

3. DeepAccident 的基础路点是匀速直行模拟，不是实际 CoDriving 模型输出。`simulate_codriving_waypoints()` 已在代码注释中说明这是近似下界，见 `experiments/run_deepaccident_unified.py:60`。因此 WPColl 证明的是安全约束对这一路点生成器的修正效果，不是端到端规划器真实输出的闭环效果。

4. DeepAccident 使用 GT perception，并默认包含不可见 agent。虽然这符合“协同感知+可见性标记”的研究设定，但比真实传感器噪声更强。需要在实验中额外报告 ego-only、coop、噪声鲁棒和漏检设置。

5. `NoCon-egoonly` 与 `NoCon-coop` 在 SUMO 中没有控制方法，coop perception 不会被使用，所以二者结果应基本相同。它们更适合作为“无控制真实碰撞率”上下文，而不是协同感知有效性的直接证据。

6. SUMO 闭环对 `constrain_waypoints()` 方法有额外路点到车道动作的映射：`apply_waypoint_lane_intent()`，见 `experiments/run_modified_sumo_comparison.py:513`。这对 UniE2EV2X、MAP、RiskMM 和 Hybrid 都生效，但 `RearEscape` 的后车逃逸逻辑只属于本文方法，比较时应作为本文改进模块而非通用接口能力。

7. SUMO 压力场景是手工确定性构造，能证明压力原因分析和改进有效，但还不能代表大规模随机交通分布。需要随机种子、车辆数量、速度分布和遮挡布局的扩展验证。

8. 完整三层 `SafetyConstraintModule` 不是当前最优结果的主要执行路径。论文如果强调 RiskMap/RiskGraph/RiskEvents，应清楚区分“代码实现的完整框架”和“当前实证最优的轻量 Hybrid 版本”。

9. V1 的 TTC head 和 Hybrid 的 `risk_model` 预留接口没有在当前最优主线中发挥核心作用。若论文写“神经 TTC”或“V2 风险打分”，需要补实验或删减表述。

10. `MinimumHarmPlanner._evaluate_harm()` 中 final speed 的计算曾使用 `trajectory[-1, 2]`，而 bicycle state 第 2 维是 heading，不是速度。该问题已在 260519 后续修复为使用 `trajectory[-1, 3]`，但完整最小伤害模块仍需要专门 ablation 后再作为强贡献报告。

11. `FeasibleRegionComputer` 和 `HierarchicalConstraint` 有重复 import，影响不大，但说明代码还有整理空间。

基于以上问题，当前不满足“代码没有问题”的前提，所以本次不生成正式论文初稿。更合适的下一步是先修复/补齐上述问题，再写论文。
