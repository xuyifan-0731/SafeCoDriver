# 260508 代码审计报告 (Repository Audit)

> **审计目的**: 对整个 SafeCoDriver 仓库做严格、批判性的代码审查，识别可能影响实验公平性、可信度和结论有效性的问题。
> **审计范围**: 核心算法 (`coop_safety/`) + 实验脚本 (`experiments/`) + 文档结论
> **审计原则**: 假设代码可能存在偶然或隐蔽的错误；不为已有结论辩护；只报告可证实的问题。

## 一、严重问题 (会影响实验结论)

### 🔴 1. 训练-评测数据泄漏 (Critical)

**文件**: `coop_safety/learned/train_collision.py:36-78`

**问题**:
```python
class DeepAccidentDataset(Dataset):
    def __init__(self, loader, split='train', train_ratio=0.8):
        self.samples = []
        for si, s in enumerate(loader.scenarios):
            for fi in range(len(s['frames'])):       # ← 帧级遍历
                ...
                self.samples.append({...})           # ← 每帧一个样本

        np.random.seed(42)
        np.random.shuffle(self.samples)              # ← 帧级 shuffle
        split_idx = int(n * train_ratio)
        if split == 'train':
            self.samples = self.samples[:split_idx]
        else:
            self.samples = self.samples[split_idx:]
```

**性质**: 数据按 **帧级别** 打乱，而不是 **场景级别**。一个场景包含 50-100 帧，相邻帧几乎相同 (位置变化 <0.5m)。打乱后训练集和验证集中含有同一场景的不同帧，构成**数据泄漏 (data leakage)**。

**进一步严重问题**: 在 `experiments/run_deepaccident_unified.py` 和 `run_final_eval.py` 中，V1 模型在**全部 104 个场景** (含 80% 用于训练的帧) 上被评测。论文报告的 "Ours-Hybrid 100% DetRate, AUC=0.985" **部分是在训练数据上测得的**。

**影响**: 
- V1 网络的真实泛化能力被严重高估
- "100% DetRate" 在训练分布上为真，对未见场景未必为真
- DeepAccident 离线对比的 Ours-Hybrid vs 其他方法不公平 (其他方法没在数据上训练过)

**修复方案**: 改为按 **场景** 划分 train/val。或：训练只用 val_part1 (52 场景)，评测在 val_part2 (52 场景)。

### 🔴 2. RSS / APF 基线在所有评测中"无效化" (Critical)

**文件**: `experiments/methods.py:129`, `experiments/methods_modern.py:133`

**问题**:
```python
# methods.py - RSSOnly.constrain():
return SafeActionSpace(
    feasible_region=coords,
    ...
    mode=ConstraintMode.NORMAL,    # ← 永远是 NORMAL
    ...
)
```

RSS 和 APF (`RiskPotentialField`) 的 `constrain()` 接口**总是返回 `mode=ConstraintMode.NORMAL`**——尽管它们计算了排除区域 (`feasible.difference(...)`)，但从未根据排除结果改变 mode。

**评测代码** (`run_deepaccident_unified.py`):
```python
safe = method.constrain(frame.perception)
was_modified = safe.mode != ConstraintMode.NORMAL  # 对 RSS 永远是 False
```

**结果**: 论文中 "RSS 在 DeepAccident 上 0% DetRate" 的结论是错误的——不是 RSS 失败了，而是**实现里根本没有连接 mode 信号**。RSS 实际上有"距离过近 → 排除区"的逻辑，但这个信号被完全忽略了。

**SUMO 实验同样受影响**: `run_final_eval.py:185-192` 中 RSS 走 `else` 分支，依赖 `safe.mode != NORMAL` 触发减速，所以 RSS 在 SUMO 中也从不触发，等同于 NoConstraint。

**修复方案**: 在 RSSOnly 中根据 `min_ttc` 或 `dist < d_safe` 设置 `mode=CONSERVATIVE`/`MINIMUM_HARM`。或论文表述改为"RSS 提供约束但无离散预警 mode"。

### 🔴 3. RiskMM 基线在 SUMO 中也无法触发 (Critical)

**文件**: `experiments/methods_new_baselines.py:383-387`

**问题**: RiskMM 的 `constrain_waypoints` 返回的 stats 字典只含 `n_modifications`，**不含 `n_collisions_detected` 也不含 `n_geometric_threats`**:

```python
stats = {
    "method": self.name,
    "n_modifications": n_modifications,
    "modification_rate": n_modifications / max(len(waypoints) * 5, 1),
}
```

**评测代码** (`run_final_eval.py:167-168`):
```python
n_threats = stats.get('n_geometric_threats', 0)   # → 0 (字段不存在)
n_det = stats.get('n_collisions_detected', 0)     # → 0 (字段不存在)
if n_threats > 0 or n_det > 0:                    # → 永不触发
    warned = True
```

**结果**: SUMO 中 RiskMM 永不减速 → 与 NoConstraint 等同。论文中 "RiskMM 在 SUMO 上 16 次碰撞 = NoConstraint" 是因接口不匹配，不是方法失败。

**修复方案**: 在 RiskMM 的 stats 中添加 `n_geometric_threats` 或 `n_collisions_detected` 字段；或在 evaluation 中按 `n_modifications > 0` 判断。

### 🟠 4. Hybrid 方法的"检测-修正解耦"创造了不公平对比 (Major)

**文件**: `coop_safety/learned/hybrid_safety.py:191-251`

**问题**: Ours-Hybrid 实际是**两个独立机制并联**:
- 路点修正: 始终基于几何检测 (visibility-aware margin) → 决定 `WPColl%`
- 碰撞预警: V1 网络 → 决定 `DetRate / FalseAlm`

**对比方法只有一个机制**:
- UniE2EV2X / MAP: 几何碰撞检测 → 同时决定路点修正和预警
- RiskMM: 风险场梯度优化 → 同时决定路点修正和预警

由于 Hybrid 用 V1 (在数据上训练过, 见问题 1) 做检测、用几何做修正，它在 `WPColl%` 和 `FalseAlm` 上都能得到最优值。这不是因为方法更好，而是因为它使用了两套独立机制，每套各擅长一项指标。

**影响**: Hybrid 报告 "WPColl=0.2%, FA=17.3%" 看起来比 MAP "WPColl=0.4%, FA=94%" 好，但这种比较忽略了：
- MAP 只用一个机制完成两项任务
- Hybrid 的低 FA 来自 V1 (训练在评测数据上)

**修复方案**: 
- a) 在论文中明确披露 "Hybrid 使用了两个独立机制"
- b) 添加消融: 只用几何机制 vs 只用 V1 vs Hybrid (这个之前做过，但应在主表强调)
- c) 用真正未见数据 (e.g., DAIR-V2X) 重新评测 V1 检测能力

## 二、中等问题 (会影响数值精度)

### 🟡 5. SUMO 场景太"安全"，区分度不够

**文件**: `experiments/run_final_eval.py` 输出

**问题**: 终版实验中 NoConstraint 在 30 个场景 × 3 seeds = 90 次运行中只产生 16 次碰撞 (~5.3 次/seed)。这意味着大多数场景对 ego 来说本来就是安全的。在如此低风险场景中：
- Ours-Hybrid 0 碰撞 vs NoConstraint 16 碰撞 看起来差异很大
- 但实际"风险窗口"很窄，结果对 SUMO 内部参数 (departure timing) 非常敏感

**佐证**: 不同 seed 的碰撞数变化很大 — t_junction `[1, 3, 2]`, crossroads `[5, 3, 2]`，方差不小。

**修复方案**:
- 显式设计高冲突场景: ego 与对向车 / 横向车在路口同时到达 (依赖固定 depart 而非随机)
- 增大场景数 (从 30 到 100+)
- 报告置信区间而非点估计

### 🟡 6. "二次碰撞率" 定义不一致

**文件**: `run_final_eval.py:284-288`

**问题**:
```python
m.unique_ego_coll += 1
if m.first_collision < 0:
    m.first_collision = step
else:
    m.secondary_coll += 1   # 第 2/3/4 次都算 1 次 secondary
```

**当前定义**: N 次唯一碰撞 → N-1 次"二次碰撞"。这定义了"非首次碰撞"。

**用户原始问法是**: "碰撞三次算两次二次碰撞" — 这等于 N - 1。一致 ✓

但文档表述含糊。例如: "二次率 = 二次碰撞 / 唯一碰撞" 在 t_junction 中得到 0/6 = 0%。这容易被误解为"无连环碰撞"，但实际上是**6 次都是初次碰撞 (与不同 agent)**。这个含义需要在论文中明确。

---

### 🟡 7. 坐标系一致性疑问 (DeepAccident)

**文件**: `coop_safety/learned/hybrid_safety.py:102-103`, `experiments/deepaccident_loader.py:111`

**问题**: agent 位置预测使用:
```python
ax = a.x + a.vx * dt
ay = a.y + a.vy * dt
```

要求 `a.vx, a.vy` 是 **ego 帧下的相对速度** (即已减去 ego 速度)。

但 `deepaccident_loader.py:111` 直接使用标签中的 `vx, vy`:
```python
state=VehicleState(... vx=a['vx'], vy=a['vy'], ...)
```

DeepAccident 标注规范不明确：`vx, vy` 是绝对速度还是 ego 相对速度？文档中没有说明，代码中也没有 normalization 步骤。

如果是绝对速度，那 `_find_threats` 的预测就忽略了 ego 自身的运动 — 在 ego 高速行驶时，agent 的实际相对位置和预测会有偏差。

**修复方案**:
- 在 `label_to_perception` 中显式减去 ego 速度: `vx = a['vx'] - ego_vx`
- 或在 DeepAccident 文档中验证标签语义后注释清楚

---

### 🟡 8. 静默 try/except 块过多

**文件**: 整个仓库 (尤其 `run_final_eval.py`, `hybrid_safety.py`)

**问题**: 大量 `except Exception: pass` 或 `except: return 0.0` 没有日志。例如:
- `hybrid_safety.py:166-167`: V1 推理失败 → 返回 0 (无危险)，但具体错误被吞掉
- `run_final_eval.py:262`: `except: mw = wp` 让模型输出默默退化
- `run_final_eval.py:289`: `except: pass` 跳过整个碰撞处理

**影响**: 
- 真实错误被掩盖，结果可能偏向"看起来正常"的方向
- 调试困难

**修复方案**: 将 except 替换为 `logging.warning(traceback.format_exc())` 或至少 `print(e)`。

### 🟢 10. 阈值不一致

- `HybridSafetyConstraint.__init__`: `detection_threshold=0.5`
- `constrain()` 方法: 使用 `> 0.7` (硬编码 MINIMUM_HARM) 和 `> self.detection_threshold` (CONSERVATIVE)
- `run_deepaccident_unified.py` 中的 `OursV1` 类: 硬编码 `> 0.7` 和 `> 0.3`

不同入口阈值不统一。

### 🟢 11. `n_collisions_detected` 字段语义混乱

在 `hybrid_safety.py:246`: 设为 0/1 (布尔)。
在 `methods_new_baselines.py:111-114` (UniE2EV2X): 设为整数计数。

字段名"_detected"暗示计数，但实际有时是布尔。

### 🟢 12. 魔法数字 `dt=0.5`, `MAX_AGENTS=30`, `time_horizon=10`

这些在多个文件中重复出现，没有集中配置。如果改了一处，可能漏改其他处。

### 🟢 13. NoConstraint 在 DeepAccident 上 WPColl=2.9%

这是 `simulate_codriving_waypoints` 用 constant-velocity 生成的"baseline 路点"的碰撞率。它**不是** CoDriving Planner 的真实输出 (CoDriving 是个端到端模型)。所以"WPColl% from NoConstraint" 的意义实际是"恒速行驶的碰撞率"，不是"无安全约束的真实驾驶碰撞率"。论文措辞应注意这一区分。

## 四、文档与结论中的偏差

### 📝 14. `260508终版实验sumo_DeepAcc.md` 中的过强结论

文档原文: "Ours-Hybrid 是唯一同时满足 [所有指标最优] 的方法"。

但基于上面发现的问题:
- DeepAccident 的"100% DetRate"含训练数据泄漏成分
- "RSS/RiskMM 16 collisions" 来自接口未连接，不是方法实质失败
- "Ours 0 collisions" 在 30 场景下成立，但场景生成偶然性高

**建议表述**:
> "在我们当前的实验设置下，Ours-Hybrid 取得最优观测值。但 V1 模块训练-评测使用同一批 DeepAccident 场景，对未见场景的泛化未经独立验证。RSS/RiskMM 在 SUMO 中的低分主要反映接口适配问题，而非算法本质失效。"

### 📝 15. AUC=0.985 在论文中报告时应注明

如果保持当前训练流程，应在论文中注明 "validation set drawn at frame level from same scenarios as training data"，并补充至少一个真实分场景划分的结果。

## 七、总结

仓库整体**研究方向和工程实现是合理的**，但存在 3 个会影响论文结论的关键问题:

1. **训练-评测数据泄漏** → V1 性能被高估
2. **RSS/APF/RiskMM 接口未正确连接** → 基线被无意削弱
3. **检测-修正解耦带来的"双机制优势"** → 需要更明确的消融和披露

如果在论文投稿前修复 P0 项 (重训 + 修复基线接口 + 跨数据集验证)，结论的可信度将显著提升。当前结论 **不应** 在论文中以"最优"形式呈现，而应描述为"在配对场景下的强结果，需要更广泛验证"。
