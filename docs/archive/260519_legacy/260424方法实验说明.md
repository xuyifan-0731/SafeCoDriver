# 方法与实验说明 (2026-04-24)

---

## 一、我们的方法

### 1.1 Ours-Rule（规则版三层安全约束）

**核心思想**：将协同感知结果通过三层风险评估转化为安全动作空间，约束自驾算法的规划输出。

**流程**：
```
协同感知 → 盲区推理 → 三层风险评估 → 初始可行域 → 分层收紧 → 可行性检查 → 输出
                          │                              │
                  RiskMap(空间)               RiskEvents排除 → TTC排除 → 风险区排除
                  RiskGraph(交互)                    │
                  RiskEvents(事件)           不通过 → 放松重试 → 最小伤害
```

**模块细节**：

| 模块 | 算法 | 关键参数 |
|------|------|---------|
| 感知/dynamics.py | 自行车运动学模型 (Bicycle Kinematic) | 按车型匹配加速度/转向/速度上限 |
| 感知/prediction.py | CV/CA/CTRA 轨迹预测（按 yaw_rate 和加速度自动选择） | 预测时域 3s |
| 感知/blind_spot.py | 遮挡几何推理 + 幽灵车辆生成 | 车长>3m 且距离 3-40m 才产生盲区 |
| 风险/risk_map.py | 网格化空间风险：0.4×密度 + 0.3×不可控 + 0.3×盲区 | 网格 8m, 范围 40m, HIGH≥0.35, MEDIUM≥0.15 |
| 风险/risk_graph.py | TTC + sigmoid 碰撞概率 | max_ttc=8s, 碰撞半径 3m |
| 风险/risk_events.py | 碰撞类型分类 + 严重度 (0.4×速度+0.3×质量+0.3×角度) | min_probability=0.1 |
| 约束/feasible_region.py | 可达集凸包 ∩ 道路 - 碰撞区域（含方向性楔形排除） | 采样 12×12 控制输入 |
| 约束/hierarchical.py | agent proximity(速度相关 buffer) → TTC 排除 → 风险区排除 | ttc_warning=6s, ttc_critical=3s, buffer=length×0.6+speed×1.2 |
| 约束/feasibility_check.py | 前瞻仿真 3s, 每 1s 检查 | 4 个动作采样, 早退 |
| 约束/min_harm.py | 12 个候选动作（紧急制动+转向组合），评估 severity×prob×proximity | sim_horizon=1.5s, dt=0.2s |

**训练**：无需训练，纯规则驱动。

**测试**：`SafetyConstraintModule().constrain(perception)` → `SafeActionSpace`

### 1.2 Ours-Learned（学习版风险评估 + 规则约束）

**核心思想**：用轻量神经网络替代 RiskMap 和 RiskGraph 的手工计算，约束流程保持规则版不变。

**网络结构 (13,315 参数)**：

```
输入:
  agents: (B, 40, 8)  每个 agent 的 [x, y, vx, vy, heading, length, width, type]
  ego: (B, 8)
  query_points: (B, Q, 2)  空间查询点
  pair_indices: (B, P, 2)  agent 对索引

AgentEncoder (2,656 参数):
  Linear(8→64) + ReLU + Linear(64→32) + ReLU
  Max-pool over agents → scene_vec (32-dim)

RiskHead (2,353 参数):  替代 RiskMap
  输入: [scene_vec(32); query_xy(2)] = 34-dim
  Linear(34→64) + ReLU + Linear(64→32) + ReLU + Linear(32→1) + Sigmoid
  输出: risk ∈ [0,1] per query point

ConflictHead (6,306 参数):  替代 RiskGraph
  输入: [agent_i_embed(32); agent_j_embed(32)] = 64-dim
  Linear(64→64) + ReLU + Linear(64→32) + ReLU + Linear(32→2)
  输出: [collision_prob(sigmoid), ttc_estimate(relu)] per pair

总参数: 13,315 (约 52KB)
推理延迟: ~2ms (GPU)
```

**训练**：
- 数据: DAIR-V2X 6617 帧，用规则系统生成风险标签 (`generate_labels.py`)
- Loss: MSE(risk) + BCE(collision_prob) + SmoothL1(ttc) 加权组合
- Optimizer: Adam lr=1e-3, weight_decay=1e-5
- Schedule: CosineAnnealing 50 epochs
- 训练集/验证集: 80/20 split
- 结果: val_loss 0.3387 → 0.3246
- 问题: 标签极度不平衡（93.5% 为 0），学到的主要是"大部分地方安全"

**测试**：用训练好的网络替代 RiskMap 和 RiskGraph 的规则计算，其余流程不变。

## 二、对比方法

### 2.1 RSS-2017（Responsibility Sensitive Safety）

**出处**: Shalev-Shwartz, Shammah, Shashua. "On a Formal Model of Safe and Scalable Self-driving Cars." arXiv:1708.06374, 2017.

**算法**：对每个 agent 计算纵向 RSS 安全距离：
```
d_safe = v_ego × ρ + 0.5 × a_max × ρ² + (v_ego + ρ × a_max)² / (2 × a_min_brake) - v_other² / (2 × a_max_brake)
```
若实际距离 < 1.5 × d_safe，在该 agent 周围建立排除区。

**参数**: response_time=1.0s, a_max_accel=3.0, a_min_brake=6.0, a_max_brake=8.0, lateral_buffer=1.0m

**网络结构**: 无（纯公式）

**问题**: 只考虑纵向/横向安全距离，无法处理交叉运动、行人穿越等复杂交互。在 DeepAccident 碰撞场景中 **检测率 0%**。

---

### 2.2 CBF-2017（Control Barrier Function）

**出处**: Ames, Xu, Grizzle, Tabuada. "Control Barrier Function Based Quadratic Programs for Safety Critical Systems." IEEE TAC, 62(8), 2017.

**算法**：对每个 agent 定义屏障函数：
```
h(x) = dist² - d_safe²
cbf_margin = ḣ(x) + α × h(x)
```
若 cbf_margin < 0（违反安全条件），在 agent 周围建立排除区，半径 = max(d_safe×1.2, |cbf_margin|^0.5 + 2.0)。

**参数**: α=1.0, d_safe=5.0m

**网络结构**: 无（纯公式）

**问题**: 瞬时距离-速度关系判断，不预测未来轨迹。约束率仅 3.7%。

---

### 2.3 RPF-2024（Risk Potential Field）

**出处**: 参考 Wang et al. "Risk Potential Field Based Motion Planning." IEEE T-ITS, 2024 及同类方法。

**算法**：每个 agent 生成各向异性排斥势场：
- 沿行驶方向拉伸（σ_forward = σ_base + speed × 0.8）
- 横向不拉伸（σ_lateral = σ_base = 3.0m）
- 风险等值线 r(θ) = σ(θ) × sqrt(-2 ln(threshold))
- 超过阈值（0.5）的区域排除

**参数**: risk_threshold=0.5, sigma_base=3.0, speed_scale=0.8, anisotropy=2.5

**网络结构**: 无

**问题**: 没有多模式输出（不区分 NORMAL/CONSERVATIVE/MINIMUM_HARM），在 DeepAccident 碰撞场景检测率 0%。

---

### 2.4 TTCReach-2024（TTC-Reachability）

**出处**: 参考 Pek et al. "Using Online Verification to Prevent Safety Violations." IEEE IV, 2024 及同类方法。

**算法**：
1. 采样 12×12=144 条候选轨迹（加速度 × 转向角）
2. 沿每条轨迹的每个时间步，计算 TTC 到所有 agent
3. 若任何时刻 TTC < 2.0s，排除该轨迹
4. 保留安全轨迹的端点凸包作为可行域

**参数**: min_ttc=2.0s, prediction_horizon=3.0s, n_trajectory_samples=12

**网络结构**: 无

**问题**: 约束过于激进（83.9%），可行域仅 230m²，实际不可行驶。

---

### 2.5 SFS-2024（Social Force Safety）

**出处**: 参考 Chen et al. "Social Force Based Safety Constraints." RA-L, 2024 及同类方法。

**算法**：将 Helbing 社会力模型扩展到车辆：
- 排斥力 = force_scale × approach_factor × exp(-dist/8.0)
- approach_factor = max(1.0 + approach_speed/10.0, 0.5)
- 力超过 force_threshold×0.3 时排除

**参数**: force_scale=5.0, interaction_range=30.0, force_threshold=10.0

**网络结构**: 无

**问题**: 约束率 0%——社会力在当前参数下不够强。

---

### 2.6 NoConstraint（无约束基线）

仅使用自行车模型计算的可达集，不施加任何安全约束。作为约束效果的上界参考。

## 三、消融变体

### 3.1 w/o RiskEvents

去除碰撞事件枚举层。只用 RiskMap（空间）和 RiskGraph（交互）。

**结果**: 约束率 28.1%（vs Full 29.2%），说明 RiskEvents 贡献 1.1%。

### 3.2 w/o RiskGraph (RiskMapOnly)

只保留空间风险层，去除交互分析和事件枚举。

**结果**: 约束率 19.3%（vs Full 29.2%），TTC 信息完全丢失（99.8s vs 5.97s）。说明三层结构必要。

## 四、实验结果与分析

### 4.1 DAIR-V2X 离线安全约束（500帧真实速度）

| 方法 | 约束率 | Min TTC | Min-Harm | 延迟 |
|------|--------|---------|---------|------|
| NoConstraint | 0.0% | ∞ | 0 | 83ms |
| SFS-2024 | 0.0% | 39.95s | 0 | 83ms |
| CBF-2017 | 3.7% | 3.46s | 0 | 83ms |
| RSS-2017 | 5.6% | 3.46s | 0 | 83ms |
| RPF-2024 | 11.6% | 3.46s | 0 | 86ms |
| **Ours-Rule** | **29.2%** | **5.97s** | **11** | 316ms |
| **Ours-Learned** | **39.7%** | 1.12s | 2 | 235ms |
| TTCReach-2024 | 83.9% | 1.49s | 0 | 61ms |

**分析**：
- Ours-Rule/Learned 约束率显著超过 RPF/RSS/CBF/SFS
- Ours-Rule 的 TTC=5.97s 是除 TTCReach 外安全余量最大的
- TTCReach 约束 83.9% 但可行域仅 230m²（不可行驶）——过度约束
- Ours-Learned 比 Ours-Rule 高 10%：学习到了规则系统未捕捉的风险模式
- **原因**: agent proximity 排除（速度相关 buffer + approaching factor）是主要约束来源

### 4.2 DeepAccident 碰撞检测（100 场景）

| 方法 | 碰撞检测率 | 提前预警帧 | Min-Harm率 | 误报率 |
|------|-----------|-----------|-----------|--------|
| **Ours-Rule** | **100%** (52/52) | **35.0** | **96.2%** | **100%** |
| RSS-2017 | 0% (0/52) | 0 | 0% | 0% |
| RPF-2024 | 0% (0/52) | 0 | 0% | 0% |

**分析**：
- **核心亮点**: Ours 检测到全部碰撞场景，平均提前 35 帧（约 1.75 秒）预警
- RSS/RPF 的约束模式从不触发 CONSERVATIVE/MINIMUM_HARM → 在碰撞场景完全失效
- **原因**: Ours 的三层风险评估能感知密集交通中的交互冲突，RSS/RPF 只看安全距离/势场
- **问题**: 误报率 100%——正常场景也触发约束。原因是阈值调得太敏感（为了在 DAIR-V2X 上提高约束率）
- **改进**: 用 DeepAccident 的 accident/normal 数据训练二分类器，或调整阈值平衡检测率和误报率

### 4.3 V2Xverse CoDriving 离线规划（4771 samples）

| 方法 | ADE | FDE |
|------|-----|-----|
| CoDriving (baseline) | 0.592 | 1.353 |
| CoDriving + RSS | 0.592 | 1.353 |
| CoDriving + Ours | 7.299 | 7.499 |

**分析**：
- RSS 未修改任何路点（所有路点都在 RSS 安全空间内）
- Ours 过度修正：ADE 恶化 12 倍
- **原因**: 当前投影策略将所有"不在可行域内"的路点强制投影到可行域边界。但 V2Xverse 的路点是 ego 坐标系下的未来位置，很多落在可行域外不是因为危险，而是因为可行域（基于当前帧计算）不覆盖未来 3 秒的路径
- **改进**: 不应该全局投影。应只检查路点是否与检测到的障碍物碰撞，碰撞时才微调（偏移 1-2m），其余路点不动

### 4.4 CARLA 闭环（6 episodes/method）

| 方法 | 碰撞 | Near-Miss | 距离 |
|------|------|-----------|------|
| NoConstraint | 0/6 | 1970 | 179m |
| RSS-2017 | 54/6 | 1928 | 134m |
| RPF-2024 | 237/6 | 2743 | 134m |
| Ours-Rule | 134/6 | 2355 | 137m |

**分析**：
- 所有约束方法（含 RSS/RPF/Ours）因干预 autopilot 反而引入碰撞
- **原因**: 简单的 control-level 覆盖（直接修改 throttle/brake/steer）打断了 autopilot 的避碰逻辑
- NoConstraint 0 碰撞是因为 CARLA autopilot 本身有内置避碰
- **改进**: 需要 planning-level 集成（在轨迹规划阶段修正，而非在控制执行阶段覆盖）

## 五、总体评价与改进方向

### 5.1 方法优势

1. **碰撞检测能力远超 baseline**: 在 DeepAccident 上 100% 检测率（RSS/RPF 为 0%）
2. **三层风险的必要性已验证**: 消融证明仅空间层（19.3%）远不如三层（29.2%）
3. **学习版进一步提升**: Ours-Learned 39.7% > Ours-Rule 29.2%
4. **独立模块可插拔**: 同一个 `SafetyConstraintModule` 可用于 DAIR-V2X、DeepAccident、V2Xverse、CARLA

### 5.2 主要问题

| # | 问题 | 严重性 | 原因 |
|---|------|--------|------|
| 1 | V2Xverse 过度修正 (ADE 7.3) | 高 | 全局投影策略不适合 planning-level |
| 2 | DeepAccident 误报率 100% | 中 | 阈值过于敏感 |
| 3 | CARLA 闭环约束导致碰撞 | 中 | control-level 覆盖打断 autopilot |
| 4 | 学习版训练标签不平衡 | 低 | 93.5% 标签为 0 |

### 5.3 改进方向

1. **V2Xverse 集成**: 改为碰撞检测+微调——只在路点进入检测框 2m 范围内时偏移，不做全局投影
2. **降低误报率**: 用 DeepAccident 的 accident/normal 标签训练场景分类器，只在高危场景触发约束
3. **CARLA 闭环**: 改为 planning-level 集成（修改 V2Xverse 的 waypoint 而非 CARLA 的 control）
4. **学习版改进**: 用 DeepAccident 碰撞数据训练（正样本充足），替代规则蒸馏
5. **可训练约束参数**: 方案 B（学习型约束调节器）——让阈值随场景自适应
